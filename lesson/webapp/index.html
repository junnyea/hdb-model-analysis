1. What are we measuring?

Regression models predict a continuous number (a price, a temperature, demand). For each row we compare the actual value yᵢ against the model's prediction ŷᵢ. These three metrics summarise how close they are.

MetricReads asUnits
MAE"Off by ±X on average"same as target
MAPE"Off by X% on average"percent
R²"Explains X% of the variance"0 → 1 (can be negative)

3. The formulas

MAE  = (1/n) · Σ |yᵢ − ŷᵢ|
MAPE = (100/n) · Σ |yᵢ − ŷᵢ| / |yᵢ|
R²   = 1 − Σ(yᵢ − ŷᵢ)² / Σ(yᵢ − ȳ)²

ȳ is the mean of the actual values. R² compares your model against a baseline that always guesses ȳ.

4. In scikit-learn

from sklearn.metrics import mean_absolute_error, \ mean_absolute_percentage_error, r2_score

mae  = mean_absolute_error(y_true, y_pred)
mape = mean_absolute_percentage_error(y_true, y_pred) * 100
r2   = r2_score(y_true, y_pred)

🌲 Random Forest vs Gradient Boosting — and overfitting

Now we keep the features fixed (the 8-feature enhanced set) and change the model. Random Forest grows ~100 trees independently and averages them. Gradient Boosting builds trees one after another, each fixing the previous trees' mistakes.

The honest test isn't just "who has the lowest error" — it's also how much does the model overfit? A model that scores far better on training flats than on unseen test flats has memorised rather than learned.

💡 So which is "better"?

After tuning, the answer becomes clear:

  • 🏆 Tuned Gradient Boosting — highest test R² (0.964), MAE tied with Random Forest (S$26.4k), and a tiny overfitting gap (+0.002). The model to ship.
  • Random Forest — marginally lower MAE (S$25.8k) but a large gap (+0.032); it buys its error by memorising.
  • Default Gradient Boosting — honest but under-powered until tuned.

The deeper lesson: "better" isn't the single lowest test number — it's a low test error and a small train↔test gap. Tuning (max_iter=500, learning_rate=0.1, l2_regularization=1.0) gave Gradient Boosting both.

🤝 Stacking — can a team of models beat the best single one?

Stacking trains several base models side by side, then a small "manager" model learns how to blend their predictions. The idea: each model makes different mistakes, so a smart blend can beat any one of them.

Part D tried it with a naïve team (Random Forest + a default Gradient Boosting) and it barely helped. Part D+ rebuilds the team to be stronger and more diverse.

Why did the enhanced team win when Part D didn't?

  • Diversity — Part D blended two tree models that make similar mistakes. Part D+ adds RidgeCV, a linear learner that errs differently — the engine of stacking.
  • Stronger bases — swapping the default booster for our tuned HistGB raises the team's ceiling.
  • passthrough=True — the manager sees the original features and the base predictions, so it can correct them with context.
  • Regularised manager — RidgeCV resists overfitting the blend itself.

🎯 The five lessons

  1. Measure honestly — always test on unseen data (train/test split); report MAE and R².
  2. Better features beat fancier models — Step 3 (features) gained more than Step 4 (model swap).
  3. Clean your raw columns — turning "61 years 04 months" into a number unlocked the biggest jump.
  4. Watch the overfitting gap — a low test error with a huge train↔test gap is a warning, not a win.
  5. "Best" = low error and small gap — tuning let Gradient Boosting win on both.
  6. Ensembles need diversity — stacking only beat the best single model when the team mixed different learners (trees + linear); and even then the gain was modest for ~3× the cost.

🧭 Overall conclusion — thinking like a data scientist

The numbers tell what happened. This is how a practitioner reasons about them — five lenses an ML specialist applies to a project like this.

1 · Spend effort where the leverage is

Error fell S$102k → S$24.6k — but where did the gains come from?

LeverDropEffort
Features (B, B+)S$102k → S$53k (~half the total)Hours of data cleaning
Model + tuning (C, C+)S$53k → S$26kMinutes of code, more compute
Ensembling (D, D+)S$26k → S$24.6k (~6%)~3× compute, more complexity

Classic diminishing returns. Juniors reach for fancier models first; seniors interrogate the data first — that's where the cheap, large wins live. A messy text column (remaining_lease) cleaned into a number beat every algorithm swap.

2 · A single metric is a trap — triangulate

We never judged on one number: MAE gives a dollar-intuition stakeholders feel, MAPE normalises across cheap and pricey flats, R² sanity-checks against a naïve mean-guess. And we paired the test score with the train↔test gap — the bias–variance lens. Random Forest's lowest MAE hid heavy overfitting (train R² 0.994 vs test 0.962). The best model on one axis can be wrong once you add the second: generalisation.

3 · "Best" is a business decision, not a leaderboard

Enhanced stacking won every accuracy metric — yet isn't automatically the right ship:

  • Latency / cost — ~3× training time and a heavier artifact for a ~6% MAE gain.
  • Maintainability — three base models + a manager is more to monitor, debug and re-train.
  • Marginal value — is S$1,800 less average error worth that to this business? For a pricing guide, probably not; for a fund trading on it, maybe.

A senior picks the model on the Pareto frontier of accuracy vs. cost / complexity / risk — here, the tuned single Gradient Boosting is often the pragmatic winner.

4 · Trust comes from honest evaluation, not high scores

Every score is on held-out data the model never saw — the only honest estimate of real-world performance. The app's ±MAE error band communicates uncertainty instead of false-precision. That's the difference between a model that looks good in a notebook and one a business can rely on.

5 · Know when to stop

The curve flattened at Part D. Chasing fractions of a percent with exotic models is low-ROI. The real next moves aren't more modelling — they're better data (renovation, distance-to-MRT, amenities), monitoring for drift (prices move with policy & rates — txn_year already mattered), and validating business impact. Modelling is one loop in a larger system.

🎯 The mindset in one line

A data scientist's job isn't to maximise a metric — it's to make a trustworthy, cost-aware decision under uncertainty. Get the data right, measure honestly on unseen data, judge accuracy and generalisation and cost together, and stop when the marginal gain no longer pays for its complexity.
